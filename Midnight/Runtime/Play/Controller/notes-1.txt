A few implementation notes

Your proposed refactor is sound, but I’d sharpen it a bit.

In serviceCpuReadyChunks

Split it into two phases each frame:

A. Retire completed uploads

check the upload fence for the frame/batch that was submitted earlier
if signaled:
mark associated slots Resident
destroy/recycle staging buffers
reset fence + command pool / command buffer for reuse

B. Build a new upload batch

drain some number of CpuReady chunks
create GPU-local buffers
allocate/fill staging
record copies into one command buffer
record any needed pipeline barriers
submit once
mark chunks Uploading

This is the important conceptual change: completion and submission are decoupled.

Add a budget

Do not upload unlimited terrain in one frame. Even with batching, you can still create hitches if you try to upload too much data at once.

Give yourself limits like:

max chunks uploaded per frame
max bytes uploaded per frame

That alone can help frame pacing a lot.

Prefer persistent upload infrastructure

If you are creating and destroying staging buffers for every single upload, that can also contribute to spikes.

A later improvement, still much smaller than transfer queues, is:

a per-frame staging ring / upload arena
persistently mapped host-visible memory
suballocate upload regions

That often gives another nice smoothness gain.

Barrier placement

If you keep using the graphics queue for upload + draw, synchronization is simpler:

transfer writes
barrier to vertex/index read
same queue submission order guarantees sequencing
